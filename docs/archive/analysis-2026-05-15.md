## Problem 1: Meal Name Language Inconsistency

**Status:** ❌ Confirmed — mixed Turkish/English meal names reach the user.

### Evidence

**Seed data is Turkish:**
- `cached_meals` table populated by 8 SQL files at `/Users/jans/Desktop/cached_meals_seed/`
- All meal names are Turkish: `Menemen`, `Karnıyarık ve Cacık`, `Mercimek Çorbası ve Bulgur Pilavı`, `Adana Kebap ve Salata`
- `cuisine` column is always `'Turkish'`

**Gemini prompt is English (no output language instruction):**
- File: `supabase/functions/gemma-generate-day/index.ts`, line 244-265
- `buildFullDayPrompt()` constructs an entirely English prompt
- Model is `gemini-3-flash-preview`
- No instruction like "Respond in Turkish" exists anywhere
- When cache is empty, Gemini returns English names (`"Oats"`, `"Chicken Salad"`)
- When cache has hits, Turkish names appear from seed data

**iOS displays names as-is (no translation):**
- File: `iOS/NutriWeek/NutriWeek/Features/Main/Views/MealPlanHomeView.swift`, line 668
- `Text(meal.name)` — directly renders the string from the API response
- No language detection, no translation, no mapping layer

**Side-effect: mealType() silent fallback:**
- File: `iOS/NutriWeek/NutriWeek/Data/Mappers/MealPlanMapper.swift`, line 90-96
- Only recognizes English lowercase types: `"breakfast"`, `"lunch"`, `"dinner"`
- Everything else silently falls back to `.snack`
- If Gemini returns `"Main Course"` or Turkish `"Ana Yemek"`, it becomes Snack

### Test fixture evidence
- File: `iOS/NutriWeek/NutriWeekTests/MealPlanMapperTests.swift`, line 9-23
- Example test data shows English name: `"Oats"`

## Problem 4: Generate Button Full Flow (Mapped End-to-End)

**Status:** ✅ Fully mapped — no issues found, documented for developer reference.

### iOS Call Chain

| Step | File | Line(s) | Description |
|------|------|---------|-------------|
| 1 | `MealPlanHomeView.swift` | 265-269 | User taps "Generate My Plan" button |
| 2 | `MealPlanHomeView.swift` | 99-103 | `.onChange(of: showGeneratePrompt)` triggers `handleGenerate()` |
| 3 | `MealPlanHomeView.swift` | 392-448 | `handleGenerate()` loops through 7 day slots, calls `repo.generateDay()` per slot in parallel batches (Mon+Tue first, then Wed-Sun) |
| 4 | `SupabaseMealPlanRepository.swift` | 82-89 | `generateDay()` → calls `gemmaService.generateDay()`, maps DTO to domain via `MealPlanMapper.dayPlan()` |
| 5 | `GemmaEdgeService.swift` | 28-41 | Constructs `GemmaGenerateDayRequest` payload, calls `edgeClient.invoke("gemma-generate-day")` |
| 6 | `EdgeFunctionClient.swift` | 53-60 | `client.functions.invoke()` sends POST to Supabase Edge Function with JWT auth |

### Edge Function Flow (`gemma-generate-day`)

| Step | Lines | Description |
|------|-------|-------------|
| POST validation | 921-967 | Method check, JSON parse, required fields validation |
| JWT validation | 831-887 | Validates auth token, gets userId (optional — falls back to client profile) |
| Profile override | 969-978 | If userId available, fetches dietary_preferences from DB and overrides client data |
| Cache assembly | 636-670, 994-1007 | Queries `cached_meals` via PostgREST, scores candidates, selects best per meal type |
| Cache hit (full day) | 1000-1007 | Returns cached meals if all 4 types found and within calorie tolerance |
| Gemini call (partial) | 1010-1057 | Cache has some meals → Gemini fills remaining types; retries on failure |
| Gemini call (full) | 1058-1072 | Cache miss → Gemini generates complete day plan |
| Validation | 436-467, 1074-1086 | Validates shape, required fields, dietary tags, calorie/macro tolerance (±50 cal, ±5g macro) |
| Final fallback | 1083-1107 | If combined cache+Gemini fails validation, retry full Gemini |
| Upsert + response | 765-810, 1109-1113 | New Gemini meals saved to `cached_meals`, usage_count updated, JSON response returned |

### Progressive Generation Pattern

iOS generates 7 days in parallel batches:
1. **Batch 1:** Monday + Tuesday (concurrent via `async let`)
2. UI updates: shows "2/7 days ready" with skeleton cards
3. **Batch 2:** Wednesday + Thursday + Friday + Saturday + Sunday (concurrent)
4. Final UI: all 7 days rendered with meal details
